# LegofyNet — AI Block-Figure Generator

![Version](https://img.shields.io/badge/Version-2.0-blue)

**LegofyNet** transforms your photos into sci-fi block-figure art using a custom **multiclass Pix2Pix** model (a conditional GAN with a class-aware PatchGAN discriminator) that runs **entirely in your browser** with TensorFlow.js. No servers, no uploads — your images never leave your device.

## What the app does

- **Generator** (`#/generator`): drag & drop a JPG/PNG (max 10 MB), pick one of the **22 supported classes**, and the model synthesizes a 64×128 block figure. Includes model download progress, a generation animation, an interactive **before/after comparison slider**, and PNG download at 4×/8× in crisp (pixel-perfect) or smooth scaling.
- **Classes** (`#/classes`, `#/classes/:id`): browsable registry of all 22 classes with category filters and per-class detail pages (including the literal one-hot vector each class feeds the model).
- **Gallery** (`#/gallery`): real outputs generated by the model.
- **About** (`#/about`): project story, architecture diagrams (generator / discriminator / combined), team and FAQ.
- **Terms** (`#/terms`): terms of service & privacy policy.

## The model

- **Architecture**: multiclass Pix2Pix — a U-Net generator conditioned on a 22-dim one-hot class vector, trained adversarially against a PatchGAN discriminator with an auxiliary class-prediction head.
- **I/O**: input image `[1, 128, 64, 3]` (normalized to [0,1]) + one-hot `[1, 22]` → 3 outputs, where **output index 2** is the generated `[128, 64, 3]` image.
- **Dataset**: 2,328 curated 128×64 PNG images, scraped, cleaned and labeled into 22 classes.
- **Weights**: `public/model/model.json` + 32 binary shards (~125 MB total), downloaded by the browser only when the generator page is opened (then cached).

⚠️ The class order in [src/data/classes.ts](src/data/classes.ts) (`CLASS_LABEL_ORDER`) **is the training order** — positions 11–16 are intentionally non-alphabetical. Never sort or reorder it; a dev-time assertion guards against drift.

```bash
npm install
npm run dev        # dev server at http://localhost:5173
```

Other scripts:

```bash
npm run typecheck  # TypeScript only
```

## Deploy to GitHub Pages

The repo ships a workflow at [.github/workflows/deploy.yml](.github/workflows/deploy.yml) that builds and deploys on every push to `main`.

**One-time setup**: in the repository settings, set **Settings → Pages → Build and deployment → Source = "GitHub Actions"**.

Because the app uses a relative base (`base: './'` in [vite.config.ts](vite.config.ts)) and hash routing, the same build works at `https://<user>.github.io/<any-repo-name>/` without configuration changes.

## Switching inference modes

The UI talks to an `InferenceEngine` interface ([src/services/inference/types.ts](src/services/inference/types.ts)) and never touches tensors directly. The active engine is chosen in [src/services/inference/index.ts](src/services/inference/index.ts) via an env variable:

```bash
# .env.local
VITE_INFERENCE_ENGINE=mock   # demo engine: no 125 MB download, simulated output
VITE_INFERENCE_ENGINE=tfjs   # (default) real in-browser inference
```

- **`tfjs`** — [src/services/inference/tfjsEngine.ts](src/services/inference/tfjsEngine.ts): real model. Lazily imports TensorFlow.js, reports download progress, warms up the GPU shaders, and runs predictions inside `tf.tidy` (no tensor leaks).
- **`mock`** — [src/services/inference/mockEngine.ts](src/services/inference/mockEngine.ts): clearly-labeled fake engine for UI development. The UI shows a "Demo mode" banner whenever it is active.
- **Remote API (future)** — implement `InferenceEngine` in a new `apiEngine.ts` (fetch to your endpoint, map responses to `GenerationResult`) and select it in the factory. No UI changes needed.

## Adding or editing classes

Class metadata lives in [src/data/classes.ts](src/data/classes.ts). Each entry:

```ts
{
  id: 'yoda',                  // slug used in routes and ?class= params
  name: 'Yoda',                // exact training label
  modelIndex: 20,              // one-hot index — must match training order
  category: 'jedi',            // jedi | sith | imperial | resistance | warrior | species
  image: 'images/classes/yoda.jpg',  // preview under public/
  description: '…',
}
```

To support a *new* class you must retrain/export the model with the extra label, add its preview image under `public/images/classes/`, and append it to both `CLASS_LABEL_ORDER` and `CHARACTER_CLASSES` (keeping indices aligned).

## Replacing the model

1. Export your Keras model with [tfjs-converter](https://www.tensorflow.org/js/guide/conversion) (layers format).
2. Replace the contents of `public/model/` (`model.json` + shards).
3. If the input geometry changes, update `MODEL_INPUT_WIDTH` / `MODEL_INPUT_HEIGHT` in [src/services/inference/imageUtils.ts](src/services/inference/imageUtils.ts).
4. If the output head changes, adjust the output index in [src/services/inference/tfjsEngine.ts](src/services/inference/tfjsEngine.ts) (currently `outputs[2]`).

## Known limitations

- The model weights (~125 MB) make the first generator visit heavy on slow connections; they are cached afterwards.
- Output resolution is fixed at 64×128 (the training resolution); the app upscales for display/download.
- Inputs are stretched to 64×128 without aspect-ratio preservation (training parity), so extreme aspect ratios distort.
- WebGL is effectively required — the CPU fallback works but is very slow, and low-memory mobile GPUs may fail to allocate the weights.
- The 22-class set is fixed unless the model is retrained.
